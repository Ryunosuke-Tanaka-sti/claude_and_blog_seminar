### 第1部: AI時代の「インプット → アウトプット」進化論（0:05-0:15）【10分】

#### 3. 龍ちゃんのストーリー: AI前 → AI後（3分）

**スライド内容**:

**AI前: インプット → アウトプットの習慣**
- 「学んだこと、どんなに小さくても作って学びがあれば → ブログ化」
- この習慣を続けた結果：**3年間で200記事以上**を執筆
- インプットとアウトプットをセットで実施
- **あれよあれよとブログが積み上がってきた**

**AI後: 新しい壁が出現**
- 技術的成長 → 検証内容が複雑化
- 「検証は早いけど、ブログを書くのが大変」に
- Notionコピペ地獄、関連ファイルの整理が困難
- **知見が散らばって、資産化できない**

**そして気づいた2つの課題**
- ❌ **課題1: 仕様書を書けない**
  - 検証が複雑になるほど、設計が必要
  - でも「完璧な仕様書を書けない」
  - 「何を書けばいいか分からない」

- ❌ **課題2: 知見が散らばる**
  - 検証と執筆が分離している
  - コードはあるけど、なぜそう決めたか忘れる
  - 次のプロジェクトで同じ失敗を繰り返す

**トーク**:
> 「僕は昔から『学んだことは必ずブログ化する』という習慣がありました。どんなに小さくても、作って学びがあればブログにする。この習慣で、あれよあれよと3年間で200記事以上が積み上がってきました。」
>
> 「でも、AI時代になって技術的に成長するほど、2つの壁にぶつかったんです。1つは『仕様書を書けない』、もう1つは『知見が散らばる』。この2つをどうにかしたい、と。」

#### 4. 実は今、仕様書ベースが主流になっている（2分）

**スライド内容**:
- AI開発の進化：2024年の大きな転換点
  - Vibe Coding（適当にプロンプト投げるだけ）→ **既に過去の話**
  - ✅ **Claude**：CLAUDE.md、Plan Modeで仕様書ベース推奨
  - ✅ **Cursor**：Composer Agentで段階的な開発フロー
  - ✅ **GitHub Copilot**：プロンプトエンジニアリング推奨

- 新しいツール・手法の登場
  - 📝 **Kiro**：AIがPRDを自動生成、仕様書ファーストのワークフロー
  - 📝 **OpenSpec**：仕様書をオープンソース化、共有可能な形式
  - 📝 **Spec Base開発**：多くの企業・個人が実践中

- **でも、問いたい**
  - 「仕様書を完璧に書ける人、どれくらいいますか？」
  - 「仕様書ベースが主流」と言われても、**多くの人が『書けない』で挫折している**
  - 「作って終わり」で、知見が散らばっている

**トーク**:
> 「そんな中、2024年はAI開発の転換点でした。Claude、Cursor、Kiro、OpenSpec…仕様書ベースが主流になっています。」
>
> 「でも、正直に聞きます。皆さん、仕様書を完璧に書けますか？僕は書けません。『仕様書ベースが主流』と言われても、多くの人が『書けない』で挫折してるんじゃないでしょうか。」

#### 5. 解決策: AIと一緒に作る × 4フェーズワークフロー（5分）

**スライド内容**:

**答えはシンプル: AIと一緒に作ればいい**
- 正直に言います：**僕も仕様書を完璧には書けません**
- でも、AIと一緒に作ることで：
  - AIが質問してくれる → 抜け漏れに気づける
  - AIが叩き台を作ってくれる → 人間はレビューするだけ
  - 実装で不足が露見 → 次回の仕様書が進化する
  - **仕様書作成自体が学びになる**

**そして、検証 → 記事化をワンセットに**
- AI前の「インプット → アウトプット」習慣を、AI時代に進化させる
- 4フェーズ開発ワークフローで「何個も得がある」

  ```
  フェーズ1: 計画（/docs/features/）
    ↓ AIと一緒に仕様書作成
    ✅ 得1: 仕様書の書き方が学べる
    ✅ 得2: AIが質問してくれるので抜け漏れに気づける

  フェーズ2: 実装（/application/）
    ↓ AI主導で高速検証
    ✅ 得3: 実装スピードが爆速
    ✅ 得4: 仕様の不足が露見 → 学びになる

  フェーズ3: 研究記録（/docs/research/）
    ↓ 知見の整理、設計思想の記録
    ✅ 得5: なぜそう決めたか、後から分かる
    ✅ 得6: 次回の仕様書が進化する

  フェーズ4: 記事化（/docs/article/）
    ↓ 調査資料から記事本文へ
    ✅ 得7: 記事執筆が爆速（8時間 → 4時間）
    ✅ 得8: 知見が資産化される
  ```

- **リポジトリ内で完結** - コードも記事も全てGit管理

**重要なマインドセット（スライド）**:
- ⭐ **完璧を目指さなくてOK！**
- 仕様書が書けなくても、AIと一緒に作ればいい
- 現時点での自分が思う「完璧」を目指す
- 不足は実装フェーズで露見する → **それが学びになる！**
- 計画 → 実装 → 研究記録 → 記事化のサイクルで成長
- **サイクルを回すたびに、仕様書の品質も、自分のスキルも上がる**

**Before / After比較（スライド）**:

| Before（AI前の手法） | After（4フェーズ開発） |
|---------------------|----------------------|
| 仕様書が書けず挫折 | AIと一緒に作れる |
| 検証だけで終わる | 検証 → 記事化がセット |
| 知見が散らばる | リポジトリに集約 |
| Notionコピペ地獄 | コードも記事もGit管理 |
| 記事執筆時間: 8時間 | 記事執筆時間: 4時間 |
| 同じ失敗を繰り返す | 知見が蓄積、進化する |

**トーク**:
> 「答えはシンプルです。仕様書もAIと一緒に作ればいい。AIが質問してくれるので、抜け漏れに気づけます。そして、検証 → 記事化をワンセットにする。」
>
> 「この4フェーズワークフローの素晴らしいところは、『何個も得がある』ってこと。仕様書作成で学べる、実装が爆速、知見が蓄積される、記事執筆も効率化。AI前の『インプット → アウトプット』習慣を、AI時代に進化させたんです。」

### 第2部: 4フェーズ開発ワークフロー（0:15-0:30）【15分】

#### 5. 4フェーズの全体像（3分）

**スライド内容**:
- フェーズ1: 計画（/docs/features/）
  - 仕様書作成
  - 技術選定
  - アーキテクチャ設計
- フェーズ2: 実装（/application/）
  - AI主導で実装
  - 人間は監視役
- フェーズ3: 研究記録（/docs/research/）
  - 知見の整理
  - 設計思想の記録
- フェーズ4: 記事化（/docs/article/）
  - 調査資料から記事本文へ
  - RAGもどき×文体補正

**ディレクトリ構造図（スライド）**:
```
/
├── docs/
│   ├── features/     → フェーズ1: 計画
│   ├── research/     → フェーズ3: 研究記録
│   └── article/      → フェーズ4: 記事化
└── application/      → フェーズ2: 実装
```

**トーク**:
> 「この4つのフェーズを順に進めることで、知見が自然に蓄積されます。検証 → 記事化をワンセットにするのがポイントです。」

#### 6. フェーズ1-2: 計画 → 実装（デモ）（7分）

**デモ内容**:
1. **CLAUDE.mdの設定**（2分）
   - 計画フェーズ用CLAUDE.md
   - 禁止事項: 実装コード記述禁止
   - 目的: 意思決定の明文化

2. **仕様書作成**（3分）
   - AIに仕様書を書いてもらう
   - 人間がレビューして意図を確認
   - 不明点をAIに質問

3. **実装フェーズ**（2分）
   - 仕様書を読み込ませて実装
   - 小規模システムの例（バックエンド2エンドポイント）
   - 30分で完成のデモ動画

**トーク**:
> 「実際にやってみます。まず、計画フェーズで仕様書を作成します。このフェーズでは、コードを一切書かせません。意思決定に集中します。」

#### 7. フェーズ3: 研究記録（2分）

**スライド内容**:
- 計画と実装の差異を分析
- 仕様漏れを特定して学ぶ
- 次回の改善につなげる

**ここがポイント（スライド）**:
- 💡 **仕様書の不足 = 学びのチャンス！**
- 実装フェーズで「あれ？これ仕様に書いてなかった」が必ず出る
- それを研究記録に残すことで、次回の仕様書が進化する
- 完璧じゃなくても、サイクルを回すことで成長する

**実例（スライド）**:
- 20件以上の研究記録を蓄積
  - aoai-chat-simple.md
  - github-api-integration.md
  - csv-preview-system.md

**トーク**:
> 「実装完了後、計画と実装を比較します。これにより、仕様書の不備を特定し、次回の改善につなげます。」
>
> 「実装フェーズで必ず『あれ？これ仕様に書いてなかった』って瞬間が来ます。それを研究記録に残すことで、次回の仕様書がレベルアップします。だから、**最初から完璧を目指さなくていい**んです！」

#### 8. 意思決定と実行の分離（3分）

**スライド内容**:
- 人間の役割（意思決定）
  - ✅ 何を作るか
  - ✅ どのような方針で開発するか
  - ✅ 技術選定とアーキテクチャ設計
- AIの役割（実行）
  - ✅ 決定された方針に従ってコード生成
  - ✅ ボイラープレートの大量生成
  - ✅ パターンに従った実装

**図解（スライド）**:
```
人間: 意思決定
  ↓ 計画ドキュメント
AI: 実行
  ↓ コード生成
成果物: 動くシステム
```

**トーク**:
> 「重要なのは、**意思決定は人間、実行はAI**という役割分担です。人間が「何を作るか」を決定し、AIが「どう実装するか」を実行します。」
