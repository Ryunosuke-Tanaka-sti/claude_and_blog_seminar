# 集客用概要（初級者向け）

## 概要（500文字版）

**学んだこと、きちんとアウトプットできていますか？**

SIOS Tech Labでは、エンジニアが日々学んだ内容を技術ブログとして共有することを推奨しています。私は3年間で200本のブログを執筆してきましたが、技術的成長とともに「検証は早いけど、ブログを書くのが大変」という課題に直面しました。

そこで登場したのが**「検証 → 記事化をワンセット」にするAI協業開発の4フェーズワークフロー**です。

**4フェーズ**：計画（仕様書作成）→ 実装（AI主導）→ 研究記録（知見整理）→ 記事化（資産化）

このワークフローを**モノレポ構成**で実現することで、記事執筆時間を8時間→4時間（50%削減）、開発時間を1週間→2日（71%短縮）に成功。Notionコピペ地獄から解放され、コードも記事もGit管理で一元管理できる環境を構築しました。

さらに、**仕様書レビューという上流工程の経験**を積むことで、実装をAIに任せても深い学びが得られます。

モノレポ構成×AI協業開発で、インプットとアウトプットを両方加速し、知見を資産化する実践ワークフローを学びませんか？

## セミナー概要

### こんな悩みはありませんか？

**開発・検証編**:
- ❌ プロンプトを投げても、意図したコードが生成されない
- ❌ AIが暴走して、変なコードを書いてくる
- ❌ 検証のレベルが上がって、複雑なシステムを作るのが大変
- ❌ 「AIに任せると自分が成長しないのでは？」と不安

**アウトプット編**:
- ❌ 学んだことをブログにまとめるのが大変になってきた
- ❌ ソースコードが増えて、コピーする手間が膨大
- ❌ 関連するファイルが増えて、まとめるのが難しい
- ❌ レビューする環境が欲しいけど、どうすればいいか分からない

### このセミナーで得られること

✅ **計画ドキュメントで検証の質を上げる**
- やりたいことベースを先に整理
- 検証の質が上がる → 結果的にアウトプットの質も上がる
- スピードと質、どちらも上げることができる
- 「AIが悪いのではなく、人間が悪い」という本質的な理解
- ⭐ **完璧を目指さなくてOK！** 現時点での自分が思う完璧を目指す
- 不足は実装フェーズで露見 → それが学びになる

✅ **仕様書レビューという新しい学びの機会**
- 実装をAIがやっても人間がやっても、仕様書作成で勉強になる
- **仕様書を作成・レビューする**ことで、ただコードを書くよりも多くの経験を積める
- 「たまにはコードを書きたい！」という気持ちがある場合も、仕様書作成という新しいスキルを習得
- コードレビューだけでなく、**仕様書レビュー**という上流工程の経験を積める

✅ **モノレポ構成×4フェーズ開発ワークフロー**
- `/docs/features/` → フェーズ1: 計画（仕様書で検証方針を明確化）
- `/application/` → フェーズ2: 実装（AI主導で高速検証）
- `/docs/research/` → フェーズ3: 研究記録（知見整理）
- `/docs/article/` → フェーズ4: 記事化（資産化）
- **リポジトリ内で検証 → 記事化をワンセット**にすることで知見が自然に蓄積
- コードも記事も全てGit管理で履歴が残る

✅ **記事執筆の効率化**
- ソースコードが自動的にリポジトリに集約
- 関連ファイルの整理が容易に
- AIレビュー環境の構築
- 記事執筆時間: 8時間 → 4時間（50%削減）

✅ **すぐに実践できるテンプレート**
- CLAUDE.md（計画フェーズ用）
- 仕様書テンプレート
- 研究記録フォーマット
- 記事執筆プロンプト集

- 関連ファイルが増えて整理が大変

**2. なぜ「検証だけ」「執筆だけ」ではダメなのか**
- インプットとアウトプットは両方大切
- 検証だけでは知見が散らばる
- 執筆だけでは品質が安定しない
- 計画ドキュメントで検証の質を上げる → 結果的にアウトプットの質も上がる

**2-2. 「AIに任せると成長しないのでは？」という不安への回答**

**2-3. 「完璧な仕様書を書けない」という不安への回答**

- 計画 → 実装 → 研究記録のサイクルで仕様書が進化していく

**A2**: Claude Code を中心に解説しますが、GitHub Copilot、Cursor 等でも応用可能なワークフローです。
